# `drone_interfaces`

Shared ROS 2 interface definitions for the UAS autonomy stack.

This package contains the message, service, and action contracts used to communicate between autonomy components. It intentionally contains **no flight logic, MAVLink code, MAVROS dependencies, planners, or ROS nodes**.

The goal is to give every package a stable, typed interface for exchanging aircraft state, mission requests, planned trajectories, safety status, payload tasks, and approved vehicle commands.

## Responsibilities

`uas_interfaces` owns:

* ROS 2 `.msg` definitions
* ROS 2 `.srv` definitions
* ROS 2 `.action` definitions
* Shared constants and enumerations embedded in messages
* Common coordinate-frame and altitude-reference conventions

`uas_interfaces` does **not** own:

* MAVROS subscriptions
* PyMAVLink commands
* Mission logic
* Planning or guidance algorithms
* Safety decision logic
* Payload SDK integration
* Launch files

## Interface Philosophy

The autonomy stack is divided into layers:

```text
MAVROS / MAVLink adapters
        ↓
Normalized UAS interfaces
        ↓
Mission, planning, guidance, safety, and payload logic
```

For example:

```text
MAVROS GlobalPositionTarget
        ↓
VehicleState
        ↓
Planner and Guidance
        ↓
ApprovedVehicleCommand
        ↓
PyMAVLink Command Gateway
```

Other packages should depend on `uas_interfaces` instead of depending directly on MAVROS or PyMAVLink message types.

## Core Messages

### `VehicleState.msg`

Represents the normalized aircraft state used by the autonomy system.

Typical fields include:

```text
std_msgs/Header header

bool connected
bool armed
string flight_mode

float64 latitude_deg
float64 longitude_deg
float64 altitude_msl_m
float64 relative_alt_m
float64 hae_m

geometry_msgs/Point position_enu_m
geometry_msgs/Vector3 velocity_enu_mps

float64 roll_rad
float64 pitch_rad
float64 yaw_rad

float64 ground_speed_mps
float64 airspeed_mps

bool gps_valid
bool ekf_valid
```

This message should be published by `uas_mavros_state_adapter`.

### `VehicleHealth.msg`

Represents health and validity checks used by the safety supervisor.

Typical fields include:

```text
std_msgs/Header header

bool telemetry_fresh
bool gps_valid
bool ekf_valid
bool battery_valid
bool home_position_valid

float32 battery_percent
float32 battery_voltage_v

string health_summary
```

### `MissionRequest.msg`

Represents a user, UI, CLI, or external software request for a mission.

Supported mission types may include:

```text
GO_TO_POINT
FOLLOW_ROUTE
LOITER_AT_POINT
OBSERVE_TARGET
SEARCH_TARGET
TRACK_TARGET
RETURN_HOME
LAND
```

A request should contain the mission intent and constraints, but not low-level flight commands.

### `MissionGoal.msg`

Represents the currently active, validated mission goal generated by the mission manager.

```text
MissionRequest
        ↓
Mission Manager validates request
        ↓
MissionGoal
        ↓
Planner
```

### `Waypoint.msg`

Represents one geographic route point.

```text
float64 latitude_deg
float64 longitude_deg
float64 relative_alt_m
float64 airspeed_mps
float64 acceptance_radius_m
```

### `Trajectory.msg`

Represents a route, loiter geometry, or time-indexed path generated by the planner.

The trajectory should answer:

> Where should the aircraft go next?

It should not contain actuator commands.

### `GuidanceCommand.msg`

Represents the real-time output of the guidance or control layer.

A `GuidanceCommand` is not a mission request such as `GO_TO_GLOBAL`,
`LOITER`, or `RETURN_HOME`. Those belong in `MissionRequest` and
`MissionGoal`.

`GuidanceCommand` expresses the current desired control or flight
setpoint needed to follow the active trajectory.

Supported command levels may include:

```text
NAVIGATION_SETPOINT
- desired course or bearing
- desired altitude
- desired airspeed or groundspeed

ATTITUDE_THRUST
- desired roll
- desired pitch
- desired yaw or yaw rate
- throttle

BODY_RATE_THRUST
- desired roll rate
- desired pitch rate
- desired yaw rate
- thrust

ACTUATOR_SETPOINT
- aileron
- elevator
- rudder
- throttle actuator command

### `ApprovedVehicleCommand.msg`

Represents the final command approved by the command arbiter.

Only `uas_pymavlink_command_gateway` should consume this message and translate it to MAVLink.

```text
GuidanceCommand
        ↓
Command Arbiter
        ↓
ApprovedVehicleCommand
        ↓
PyMAVLink Command Gateway
        ↓
Flight Controller
```

### `SafetyStatus.msg`

Represents the current safety state.

Typical fields:

```text
bool safety_clear
bool autonomy_permitted
bool guided_mode_required
bool rtl_required
bool hold_required

string reason
```

The command arbiter and PyMAVLink gateway should both enforce this status.

### `TargetEstimate.msg`

Represents an approximate target location supplied by an operator, sensor, or external system.

```text
string target_id

float64 latitude_deg
float64 longitude_deg
float64 altitude_m

float64 uncertainty_radius_m
float64 confidence

geometry_msgs/Vector3 velocity_enu_mps

builtin_interfaces/Time observation_time
string source
```

### `GeoObservation.msg`

Represents a single sensor-derived geolocation observation before full tracking and fusion.

Examples include a camera-derived ground estimate, range-assisted sensor estimate, or terrain-intersection estimate.

### `TargetTrack.msg`

Represents a filtered and persistent target track.

```text
string track_id

float64 latitude_deg
float64 longitude_deg
float64 altitude_m

geometry_msgs/Vector3 velocity_enu_mps
float64 uncertainty_radius_m
float64 confidence

uint8 state
builtin_interfaces/Time last_update_time
```

Suggested states:

```text
TENTATIVE
CONFIRMED
STALE
LOST
```

### Payload Messages

Payload interfaces remain separate from aircraft guidance.

```text
PayloadTask
PayloadStatus
PayloadHealth
PayloadCommand
PayloadFeedback
```

The mission manager decides that payload activity is needed. The payload manager sequences the task. The payload bridge communicates with the actual sensor or vendor SDK.

## Services

### `EnableAutonomy.srv`

Enables autonomy authority after the operator has placed the aircraft in Guided mode.

```text
---
bool accepted
string message
```

### `DisableAutonomy.srv`

Disables autonomy authority and triggers the configured safe transition behavior.

```text
---
bool accepted
string message
```

### `ResumeAutonomy.srv`

Allows deliberate resumption after autonomy was paused, such as when the operator temporarily exits Guided mode.

```text
---
bool accepted
string message
```

### `AbortMission.srv`

Stops the active mission and transitions to a configured fallback behavior.

```text
---
bool accepted
string message
```

## Actions

### `ExecuteMission.action`

Used for longer-running missions that require progress feedback and cancellation.

```text
# Goal
MissionRequest request

---
# Result
bool success
string message

---
# Feedback
string mission_state
float32 progress_percent
```

### `ExecutePayloadTask.action`

Used for long-running payload operations such as imagery collection, scanning, or sensor tasks.

## Coordinate and Altitude Conventions

All packages using these interfaces must follow the same conventions.

### Geographic Coordinates

```text
Latitude: degrees, WGS84
Longitude: degrees, WGS84
```

### Local Cartesian Coordinates

Use ROS-standard ENU for internal ROS 2 calculations:

```text
X: East
Y: North
Z: Up
```

Convert to MAVLink NED only inside the PyMAVLink command gateway when needed.

### Altitude

Never use a field named only `altitude_m`.

Use one of:

```text
altitude_msl_m
relative_alt_m
hae_m
```

Every command and message must make its altitude reference explicit.

## Dependency Rules

```text
uas_interfaces
    ├── contains ROS message, service, and action definitions only
    ├── may depend on standard ROS message packages
    └── must not depend on MAVROS, PyMAVLink, or autonomy packages
```

Recommended dependency direction:

```text
uas_interfaces
        ↑
uas_core
        ↑
ROS node packages
        ↑
uas_bringup
```

## Build

From the workspace root:

```bash
colcon build --packages-select uas_interfaces --symlink-install
source install/setup.bash
```

Inspect a generated message:

```bash
ros2 interface show uas_interfaces/msg/VehicleState
```

Inspect a service:

```bash
ros2 interface show uas_interfaces/srv/EnableAutonomy
```

## Versioning Guidance

Treat interface changes as deliberate API changes.

When changing an interface:

1. Update the corresponding `.msg`, `.srv`, or `.action` file.
2. Update all producing and consuming packages.
3. Rebuild the workspace.
4. Test message compatibility in SITL before hardware testing.
5. Avoid renaming or changing field meanings without updating documentation.

## Design Rule

> `uas_interfaces` defines what packages say to each other. It does not decide what the aircraft should do or how the aircraft is commanded.
